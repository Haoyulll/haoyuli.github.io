Data Streams

Posted on 2022-11-25 Edited on 2022-11-27

Data Streams

  • DBMS:

    • Data stored in finite, persistent data sets
    • stored sets of relatively static records
    • Good for app that require persistent data storage and complex querying

  • DSMS(Data Streams):

    • Distributed, continuous, unbounded, rapid, (real time data)
    • support on-line analysis of rapidly changing data streams

  • Comparison of DBMS and DSMS

    • DBMS: data goes through Disk
    • DSMS: Data only goes through memory.

Sampling Data Streams

Since we cannot store the entire stream, we can try to store a sample

Two different problems:

  • Sample a fixed proportion of elements in the stream (say 1 in 10)
    • As the stream grows the sample also gets bigger
  • Maintain a random sample of fixed size over a potentially infinite stream
    • As the stream grows, the sample is of fixed size.
    • At any “time” t we would like a random sample of s elements
      • What is the property of the sample we want to maintain?
        • For all time steps t, each of t elements seen so far has equal probability of being sampled

Sampling a fixed proportion 等比例

Question: How often did a user run the same query in a single days

Same as: What fraction of queries by an average search engine user are duplicates

  • Suppose each user issues x queries once and d queries twice (total: x + 2d) queries
  • x + d unque queries

Correct Answer: d/(x+d)

Proposed solution:

Distribution of Correct answer and proposed solution is not the same.

Solution of doing correct sampling:

  • Pick 1/10th of users and take all their searches in the sample
  • Use a hash function that hashes the username or user id uniformly into 10 buckets
    • We hash each username to one of ten buckets, 0 through 9
    • if the user hashes to bucked 0, then accept this search query for the sample. Discard other buckets

Generalized Problem and Solution

  • Problem: Give a data stream, take a sample of fraction a/b

  • Stream of tuples with keys:

    • Key is some subset of each tuple’s components

    • Choices of key depends on application

    • To get a sample of a/b fraction of the stream

      • Hash each tuple’s key uniformly into b buckets
      • Pick the tuple of its hash value is at most a

Sample Operator in Spark

Sample(withReplacement, fraction, seed)

  • Return a sampled subset of this RDD
  • withReplacement: can elements be sampled multiple times
  • fraction: expected size of the sample as a fraction of this RDD’s size without replacement
    • This is not guaranteed to provide exactluy the fraction specified of the total count of the given
    • Because we are using hashing techniques

Maintaining a Fixed-size Sample

Algorithm (Reservoir Sampling)

probability is calculated by hash table

  • Store all the first s element of the stream to S
  • Suppose we have seen n-1 elements, and now the nth element arrives (n > s)
    • With probability s/n (using hash table to calculate this one), keep the nth element, else discard it
    • If we picked the nth element then it replaced one of the s elements in the sample S, picked uniformly/at random

Claim

The algorithm maintains a sample S with the desired property

  • After n elements, the sample contains each element seen so far with probability s/n

takeSample Operator In Spark

takeSample(withReplacement, num, seed=None)

  • Return a fixed-size sampled subset of this RDD
  • withReplacement: can elements be sampled multiple times
  • Num: sample size
  • This method should only be used if the resulting array is expected to be small

Query streams

Sliding windows

  • window of length N - the N most recent element received
  • Interesting case: N is so large that the data cannot be stored in memory, or even on disk
  • Amazon example:
    • For every product X we keep 0/1 stream of whether that product was sold in the n-th transaction
    • We want answer queries, how many times have we sold X in the last K sales.

Counting Bits

Problem:

  • Given a stream of 0s and 1s
  • Be prepared to answer queries of the form: How many 1s are in the last k bits? where k<=N.

1.Naive Solution:

  • Store the most recent N bits
    • When new bit comes in, discard the N+1 bit

2. Simple Solution:

  • Uniformity Assumption:
    • Maintain 2 counters:
      • S:number of 1s from the beginning of the stream
      • Z:number of 0s from the beginning of the stream
    • How many 1s are in the last N bits? N*(S/S+Z)

3. Solution: DGIM

  • If N = 2^16 (64 KB), log(log N) = log(16) = 4

    • region size is log(N)

  • Solution gives approximate answer, error is never larger than 50%

  • there is tradeoff between space and accuracy

4. Solution: Fixup DGIM

Instead of summarising fix-length blocks, summarize blocks with specific number of 1s

5.Let the block sizes(number of 1s) increase exponentially

When there are few 1s in the window, block sizes stay small, so errors are small

DGIM: Timestamps

Record timestamps modulo N (the window size), so we can represent any relevant timestamp in O(log2N) bits

  • given the windows size 40(N), timestamp 123 will be recorded as 2

DGIM: Buckets

Representing a Stream by Buckets

  • The right end of a bucket is always a position with a 1
  • Every position with a 1 in some bucket
  • Either one or two buckets with the same power-of-2 number of 1s
  • Buckets do not have overlap in timestamps
  • Buckets are sorted by size
  • Buckets disappear when their end-time is > N time units in the past.

Filtering Data Streams

First Cut Solution

  • Given a set of keys S that we want to filter
  • Create a bit array B of n bits, initially all 0s
  • Choose a hash function h with range [0, n)
  • Hash each member of s in S to onoe of n buckets, and set that bit to 1. B[h(s)] = 1
  • Output value
    • Output a if B[h(a)] == 1
  • Create false positives but no false negatives
    • If the item is in S we surely output it, if not we may still output it
